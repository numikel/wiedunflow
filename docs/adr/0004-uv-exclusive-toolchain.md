# ADR-0004: UV as exclusive Python toolchain

- Status: Accepted
- Date: 2026-04-20
- Deciders: Michał Kamiński
- Related PRD: v0.1.2-draft
- Supersedes: none

## Context

Python packaging tooling in 2026 remains fragmented. The ecosystem offers pip+pip-tools, Poetry, Hatch, PDM, Rye, and UV — each with different dependency resolvers, lock-file formats, build backends, and installation semantics.

WiedunFlow targets mid-level Python developers (FR-05: "developers familiar with Python, wanting to deepen knowledge of unfamiliar repos"). This audience likely has multiple tools installed: some may use Poetry on one project, pip+pipx on another, UV on a third. When they clone WiedunFlow and follow the README, they should get a predictable, reproducible setup experience without first deciding which toolchain to use.

Historical friction:
- `pip install -e .[dev]` uses a different resolver than `poetry install` — same lockfile line can resolve to different transitive versions.
- `pip-compile` vs `poetry lock` — incompatible lock formats, no migration path.
- `pipx install wiedunflow` works, but `pip install --user wiedunflow` sometimes conflicts with system packages.
- Mixing tools in a single repo (e.g., Hatch for build, pip for CI) is confusing.

UV (Astral/Werke, written in Rust, single-file binary, <50 MB) emerged in 2024 as a modern alternative. Benefits:

1. **Correctness** — UV's resolver is more faithful to PEP 508/PEP 518 than pip's. Cross-platform behavior is deterministic (important for codecommit hash validation in PRs).
2. **Speed** — dependency resolution, installation, and invocation are typically 10–100× faster than pip/Poetry.
3. **Single binary** — one tool handles `uv sync`, `uv run`, `uv build`, `uv publish`. No version conflicts between plugins.
4. **Adoption curve** — growing adoption in high-velocity Python projects (Ruff itself uses UV, Anthropic SDK uses UV in CI, OpenAI SDK uses UV).

A written policy (ADR) prevents accidental regression to pip+pipx or Poetry when a contributor proposes "just use pip, it's simpler."

## Decision

UV is the **only** Python toolchain for WiedunFlow development, testing, and deployment:

### For Developers

- **Setup**: `curl -LsSf https://astral.sh/uv/install.sh | sh` (Linux/macOS) or PowerShell equivalent.
- **Dependencies**: `uv sync` (replaces `pip install -e .[dev]`).
- **Running tests**: `uv run pytest` (replaces `python -m pytest` after manual venv).
- **Running wiedunflow locally**: `uv run wiedun-flow --version` (replaces `python -m wiedunflow.cli`).
- **Building wheels for testing**: `uv build` (replaces `python -m build`).

### For End-Users (PyPI Release)

- **Installation**: `pip install wiedunflow` still works (our wheel is standard). Users are not forced to use UV — we are.
- **Alternative**: `uvx --from git+https://github.com/user/wiedunflow wiedun-flow init` for installation from Git (no local clone needed).

### For CI/CD

- **GitHub Actions**: use `astral-sh/setup-uv@v7` (managed action).
- **Test matrix**: Python 3.11 / 3.12 / 3.13 × Ubuntu / Windows / macOS with `uv run pytest`.
- **Lock file**: `uv.lock` is committed to the repo (platform-aware, generated by `uv sync --locked`).
- **Build & publish**: `uv build` (generates wheel + sdist), `uv publish` (OIDC trusted publishing to PyPI).

### Configuration

- **`pyproject.toml`**: single source of truth for dependencies, scripts, and metadata.
  - `[project]` — name, version, description, dependencies, optional-dependencies.
  - `[project.scripts]` — CLI entry point (e.g., `wiedunflow = "wiedunflow.cli:main"`).
  - `[tool.uv]` — optional: concurrency, index preference, but keep minimal.
  - Build backend: `[build-system] requires = ["hatchling"]` and `build-backend = "hatchling.build"`. Hatchling is passive; UV invokes it.

- **`.python-version`**: optional, pins Python version (e.g., `3.11.0`). If absent, UV defaults to the latest compatible version in the matrix.

- **`uv.lock`**: auto-generated, committed to repo. Never edited by hand.

- **No `requirements.txt`**: it will not be created. If a contributor tries to use `pip`, they will fail, and the error message directs them to `uv sync`.

### Prohibited Tools

The following are explicitly forbidden in WiedunFlow repos:

- ❌ `pip install` (use `uv sync` instead)
- ❌ `pipx` (use `uvx` instead)
- ❌ `poetry` (use `uv` instead)
- ❌ `pdm`, `rye`, `hatch` CLI (build backend is Hatchling, but do not invoke `hatch` manually; use `uv build`)
- ❌ `conda`, `mamba` (not applicable to a pure-Python CLI tool)
- ❌ Manual `venv` activation (`python -m venv` is not invoked; UV manages the venv transparently)

Any PR that introduces `requirements.txt`, `poetry.lock`, or Hatch CLI invocations will be rejected with a pointer to this ADR.

## Consequences

### Positive

- **Single source of truth** — `pyproject.toml` is the canonical dependency specification. No split between `[tool.poetry.dependencies]` and `[project.dependencies]`.
- **Cross-platform consistency** — `uv.lock` is platform-aware; a developer on Windows and one on Linux get identical transitive versions (important for reproducing CI failures).
- **Faster CI** — `uv sync --locked` is typically 30–60 seconds even with full dependency resolution; Poetry and pip often exceed 2–3 minutes.
- **Simplified onboarding** — mid-level developers see "install UV, then `uv sync`" — no toolchain selection paralysis.
- **Native Git dependencies** — `uv pip install git+https://...` works out of the box without plugins.
- **Release automation** — `uv build && uv publish` is a two-step release (vs. Poetry's three-step dance with `poetry version` → build → publish). OIDC trusted publishing is supported natively.

### Negative

- **New dependency** — developers must install UV. Not installed by default on most systems.
  - *Mitigation*: UV's installer is a single curl/PowerShell command, cross-platform, and installs a static binary (no compilation needed). Barrier to entry is minimal.
- **Ecosystem adoption** — not all Python developers are familiar with UV yet (adoption is rapidly growing in 2026, but some teams still use Poetry). Documentation must be extra clear.
- **Migration cost** — if a contributor has a Poetry `lock` file from another project, they cannot copy it to WiedunFlow. They must re-sync.
  - *Mitigation*: One-time cost. After sync, workflows are identical across projects using UV.

### Neutral

- **End-user installation** — users installing from PyPI via `pip install wiedunflow` are unaffected. Our wheel is a standard Python distribution.
- **`uvx` invocation** — users can test pre-release versions with `uvx --from git+...` without cloning the repo, but this is optional.

## Alternatives Considered

1. **Poetry + Hatchling** — rejected because `[tool.poetry.dependencies]` and `[project.dependencies]` become two sources of truth. Cross-platform lock consistency is worse than UV's.
2. **pip + pip-tools** — rejected because pip's resolver is less correct than UV's, and pip-tools adds a separate `requirements.txt` layer that diverges from `pyproject.toml`.
3. **Hatch CLI** — rejected. Hatch is primarily a build backend (which we use via Hatchling). Using `hatch` CLI for environment and build is vendor lock-in with less speed benefit than UV.
4. **Hybrid (UV dev + pip runtime)** — rejected because it fragments documentation. Developers see `uv sync` in docs but end-users might be tempted to `pip install -e .[dev]` locally, leading to dependency confusion.
5. **Multiple toolchains with compatibility matrix** — rejected because it doubles CI matrix size and maintenance burden without benefit for a single-language, single-package repo.

## Migration Criteria (reconsidering UV in v2+)

Revisit this decision if **any** of the following becomes true:

- UV's project halts maintenance or transfers to unmaintained status (as of 2026-04-20, UV is actively maintained by Astral and widely adopted).
- A competing tool demonstrates better cross-platform determinism or significantly faster performance (as of 2026-04-20, no such competitor exists).
- WiedunFlow grows beyond a single package into a monorepo that would benefit from workspace tooling (e.g., Rye workspaces). A new ADR would supersede this one.

Until then, UV remains exclusive.

## Implementation Notes

- **README.md** — include a one-line UV setup and `uv sync` instruction in the Getting Started section.
- **CI configuration** — GitHub Actions workflow should use `astral-sh/setup-uv@v7` with the latest stable version.
- **Pre-commit hooks** — configure Ruff (which is installed via UV) to run on `uv run` invocations.
- **Release checklist** — before tagging `v*.*.*`, verify `uv build && uv publish --dry-run`.
